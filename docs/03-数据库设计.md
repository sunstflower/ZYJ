# 数据库设计

> 项目：健身打卡系统（JSA）
> 数据库：MySQL 8.x，字符集 utf8mb4
> 文档版本：v1.0
> 最后更新：2026-06-16

## 1. 设计概述

数据库共设计 **3 张表**，满足"至少 2 张关联表、体现一对多关系、含外键约束"的要求：

| 表名 | 中文名 | 说明 |
|------|--------|------|
| `user` | 用户表 | 存储普通用户与管理员，通过 `role` 区分角色 |
| `sport` | 运动项目表 | 系统预置的健身项目 |
| `checkin_record` | 打卡记录表 | 核心业务表，通过外键关联 `user` 与 `sport` |

### 关系说明（一对多）

```
user (1) ────< (N) checkin_record >──── (1) sport
```

- 一个用户可以有多条打卡记录：`user 1 : N checkin_record`
- 一个运动项目可以出现在多条打卡记录中：`sport 1 : N checkin_record`
- `checkin_record` 通过 `user_id`、`sport_id` 两个外键关联，体现外键约束。

## 3. 表结构详细设计

### 3.1 user（用户表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| username | VARCHAR(50) | NOT NULL, UNIQUE | 登录用户名 |
| password | VARCHAR(100) | NOT NULL | 密码（演示阶段可明文或简单加密，见备注） |
| nickname | VARCHAR(50) | NOT NULL | 昵称/显示名 |
| role | VARCHAR(20) | NOT NULL, DEFAULT 'USER' | 角色：`USER` / `ADMIN` |
| create_time | DATETIME | NOT NULL | 创建时间 |

> **密码备注**：演示项目可先存明文以便讲解；如需提升，建议使用 BCrypt 加盐哈希。本表长度 100 已为哈希预留空间。

### 3.2 sport（运动项目表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| name | VARCHAR(50) | NOT NULL, UNIQUE | 项目名称，如"跑步" |
| description | VARCHAR(200) | NULL | 项目描述 |
| create_time | DATETIME | NOT NULL | 创建时间 |

### 3.3 checkin_record（打卡记录表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| user_id | BIGINT | NOT NULL, FK → user(id) | 打卡用户 |
| sport_id | BIGINT | NOT NULL, FK → sport(id) | 运动项目 |
| content | VARCHAR(500) | NOT NULL | 打卡文字内容 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'PENDING' | 审核状态：`PENDING`/`APPROVED`/`REJECTED` |
| checkin_time | DATETIME | NOT NULL | 打卡时间（服务端生成） |
| reviewer_id | BIGINT | NULL, FK → user(id) | 审核管理员 id（未审核为 NULL） |
| review_time | DATETIME | NULL | 审核时间（未审核为 NULL） |

**外键约束：**
- `fk_record_user`：`checkin_record.user_id` → `user.id`
- `fk_record_sport`：`checkin_record.sport_id` → `sport.id`
- `fk_record_reviewer`：`checkin_record.reviewer_id` → `user.id`

## 4. 枚举值约定

| 字段 | 取值 | 含义 |
|------|------|------|
| user.role | `USER` | 普通用户 |
| user.role | `ADMIN` | 管理员 |
| checkin_record.status | `PENDING` | 待审核（默认） |
| checkin_record.status | `APPROVED` | 审核通过 |
| checkin_record.status | `REJECTED` | 审核驳回 |

## 5. 初始化脚本 init.sql（参考）

> 该脚本将放在后端 `src/main/resources/db/init.sql`，用于在 Windows 演示机上快速重建数据库与测试数据。**实现阶段最终以代码仓库中的脚本为准。**

```sql
-- 创建数据库
CREATE DATABASE IF NOT EXISTS jsa DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE jsa;

-- 用户表
CREATE TABLE user (
  id          BIGINT       NOT NULL AUTO_INCREMENT,
  username    VARCHAR(50)  NOT NULL,
  password    VARCHAR(100) NOT NULL,
  nickname    VARCHAR(50)  NOT NULL,
  role        VARCHAR(20)  NOT NULL DEFAULT 'USER',
  create_time DATETIME     NOT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_user_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 运动项目表
CREATE TABLE sport (
  id          BIGINT       NOT NULL AUTO_INCREMENT,
  name        VARCHAR(50)  NOT NULL,
  description VARCHAR(200) NULL,
  create_time DATETIME     NOT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_sport_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 打卡记录表
CREATE TABLE checkin_record (
  id           BIGINT       NOT NULL AUTO_INCREMENT,
  user_id      BIGINT       NOT NULL,
  sport_id     BIGINT       NOT NULL,
  content      VARCHAR(500) NOT NULL,
  status       VARCHAR(20)  NOT NULL DEFAULT 'PENDING',
  checkin_time DATETIME     NOT NULL,
  reviewer_id  BIGINT       NULL,
  review_time  DATETIME     NULL,
  PRIMARY KEY (id),
  KEY idx_record_user (user_id),
  KEY idx_record_sport (sport_id),
  CONSTRAINT fk_record_user     FOREIGN KEY (user_id)     REFERENCES user(id),
  CONSTRAINT fk_record_sport    FOREIGN KEY (sport_id)    REFERENCES sport(id),
  CONSTRAINT fk_record_reviewer FOREIGN KEY (reviewer_id) REFERENCES user(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 初始化数据：用户（演示密码明文，便于讲解）
INSERT INTO user (username, password, nickname, role, create_time) VALUES
  ('admin', '123456', '系统管理员', 'ADMIN', NOW()),
  ('alice', '123456', '爱丽丝',     'USER',  NOW()),
  ('bob',   '123456', '鲍勃',       'USER',  NOW());

-- 初始化数据：运动项目
INSERT INTO sport (name, description, create_time) VALUES
  ('跑步',     '有氧运动，建议每次 30 分钟以上', NOW()),
  ('游泳',     '全身性有氧运动',                 NOW()),
  ('力量训练', '哑铃、杠铃等器械训练',           NOW()),
  ('瑜伽',     '柔韧性与核心训练',               NOW()),
  ('骑行',     '室内外有氧骑行',                 NOW());

-- 初始化数据：打卡记录（覆盖三种状态）
INSERT INTO checkin_record (user_id, sport_id, content, status, checkin_time, reviewer_id, review_time) VALUES
  (2, 1, '晨跑 5 公里，状态不错', 'APPROVED', NOW(), 1, NOW()),
```

## 6. 设计说明与权衡

1. **为什么单独建 `sport` 表**：运动项目作为独立实体，既体现外键约束与一对多关系（满足作业要求），也便于前端做下拉选择、保证数据规范。
2. **`reviewer_id` 也指向 `user` 表**：管理员本身是 `user` 表中 `role=ADMIN` 的记录，无需单独的管理员表，体现表的复用。
3. **是否启用物理外键**：MySQL InnoDB 物理外键能展示约束效果，符合作业要求，故启用。若演示中频繁清库导致顺序问题，删除时注意先删 `checkin_record`。
4. **时间字段由服务端生成**：`checkin_time` / `review_time` 在 Service 层写入，避免前端篡改。
