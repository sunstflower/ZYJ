## 2. ⚠️ 重点风险与统一建议

本机环境普遍偏「新」，而 Windows 演示机大概率装的是更主流/更旧的 LTS 版本。**核心策略：两端锁定相同的、稳妥的版本，而不是各用各的最新版。**

### 2.1 JDK 版本（最高优先级）

**现状问题：**
- 本机 `java` = 24，Maven 跑在 23，`JAVA_HOME` 未设置——三者不统一，容易出现「IDE/命令行用一个版本、Maven 用另一个版本」的诡异问题。
- JDK 23 / 24 都是短期版本（非 LTS）。Spring Boot 3.x 的官方基线与广泛验证集中在 **JDK 17 / 21（LTS）**。用 24 编译运行可能遇到依赖字节码版本、注解处理器、个别库不兼容的问题，且 Windows 演示机不一定装得到 24。

**建议（务必两端一致）：**
1. **统一锁定 JDK 17（LTS）** 作为本项目目标版本（Spring Boot 3 的最低基线，最稳、Windows 最易安装）。如团队倾向新特性，可统一用 21，但**不要用 23/24**。
2. 在 `pom.xml` 中显式声明：
   ```xml
   <properties>
     <java.version>17</java.version>
     <maven.compiler.source>17</maven.compiler.source>
     <maven.compiler.target>17</maven.compiler.target>
   </properties>
   ```
3. Mac 上安装一个 JDK 17 并设置 `JAVA_HOME` 指向它（推荐用 `jenv` 或 Homebrew 管理多版本）：
   ```bash
   brew install openjdk@17
   # 在 ~/.zshrc 中：
   export JAVA_HOME=/opt/homebrew/opt/openjdk@17
   export PATH="$JAVA_HOME/bin:$PATH"
   ```
   重开终端后 `java -version` 与 `mvn -version` 都应显示 17。
4. Windows 演示机同样安装 **JDK 17**（Temurin/Adoptium 或 Oracle JDK 均可），并设置 `JAVA_HOME` + 把 `%JAVA_HOME%\bin` 加入 `Path`。
5. 产物 `jar` 跨平台无差异：只要两端 JDK 大版本一致（都是 17），Mac 上 `mvn package` 出的 jar 在 Windows `java -jar` 直接跑。

> 备注：JDK **运行**只需大版本兼容（高版本 JVM 能跑低版本编译的字节码）；但为避免意外，演示机 JDK 版本 **≥** 编译用的 JDK 版本，且都建议就是 17。

### 2.2 MySQL 版本（次高优先级）

**现状问题：** 本机是 MySQL **9.2.0**，而 Windows 演示机/同学机器多为 **8.0.x**（甚至 5.7）。

**注意事项：**
1. **SQL 写法保持兼容 8.0**：`init.sql` 只用 8.0 也支持的语法（本项目的建表语句均为通用语法，无 9.x 专有特性），保证在 8.0 上也能执行。**不要使用 MySQL 9 新增的专有特性。**
2. **认证插件差异**：MySQL 8.0+ 默认认证插件是 `caching_sha2_password`。
   - 需使用 **Connector/J 8.0.9+**（本项目用 `mysql-connector-j` 8.x，已满足）。
   - 若演示机连接报 `Public Key Retrieval is not allowed`，在 JDBC URL 加 `allowPublicKeyRetrieval=true`（仅本地演示可用，生产勿用）。
3. **字符集统一 utf8mb4**：建库 `CHARACTER SET utf8mb4`，JDBC URL 带 `characterEncoding=utf8`，避免中文乱码。
4. **时区**：JDBC URL 必须带 `serverTimezone=Asia/Shanghai`，否则 8.0/9.0 常报 `The server time zone value ... is unrecognized`。
5. **数据不可直接拷贝**：不要把 Mac 的 MySQL data 目录拷到 Windows。统一用 **`init.sql`** 在 Windows 重新建库建表+灌测试数据。
6. **驱动依赖坐标**：用新坐标 `com.mysql:mysql-connector-j`（旧的 `mysql:mysql-connector-java` 已废弃）。

**推荐的 JDBC URL（写入 `application.yml`，两端通用）：**
```
jdbc:mysql://localhost:3306/jsa?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai&useSSL=false&allowPublicKeyRetrieval=true
```

### 2.3 CPU 架构（arm64 → x86_64）

- **Java：无影响**。JVM 屏蔽了 CPU 架构差异，arm64 上打的 jar 在 x86_64 Windows 正常运行。
- **前端构建产物：无影响**。`npm run build` 出的是纯静态文件（HTML/JS/CSS），与架构无关。
- **唯一可能踩坑**：前端若用到带 native 二进制的依赖（如某些旧版 `node-sass`），换机器需重新 `npm install` 重建。本项目用 Tailwind（纯 JS），基本无此问题。**约定：不要把 `node_modules` 拷到 Windows，到 Windows 后重新 `npm install`。**

### 2.4 Node.js 版本

- 本机 Node **v25**（很新）。Windows 演示机若用同源托管方案（前端构建产物并入后端 jar），**演示机可以完全不装 Node**。
- 若 Windows 上仍需构建/运行前端，建议两端统一到 Node **LTS（20 或 22）**，避免 v25 这种当前版在某些工具链上的不稳定。约定用 `.nvmrc` 固定版本。

## 3. 换行符 / 编码 / 路径（代码层面的跨平台约定）

| 事项 | 约定 |
|------|------|
| 换行符 | 仓库根加 `.gitattributes`：`* text=auto`，必要时 `*.sh text eol=lf`、`*.bat text eol=crlf`，避免 Mac(LF)/Windows(CRLF) 混乱 |
| 文件编码 | 全部源码、SQL、配置统一 **UTF-8**；`application.yml` 与 `init.sql` 不要存成 GBK |
| 文件路径 | 代码中不硬编码 `/` 或 `\`，用相对路径或 `File.separator`、`Paths.get()` |
| 大小写 | Mac 文件系统默认不区分大小写，**类名/文件名/import 严格大小写一致**（防止将来上 Linux 服务器出问题） |
| 端口 | 后端 8080、MySQL 3306、前端 dev 5173 写入配置，演示机被占用时只改配置 |

## 4. 演示机（Windows）环境准备清单

演示前在 Windows 上确认：

- [ ] 安装 **JDK 17**，`java -version` 显示 17，已设 `JAVA_HOME` 与 `Path`。
- [ ] 安装 **MySQL 8.0+**，记下 root 密码与端口（默认 3306），服务已启动。
- [ ] 用 MySQL 客户端执行 `init.sql` 建库建表 + 测试数据。
- [ ] 拿到后端 `jsa.jar`（已内嵌前端静态资源），按需修改随包/外置的 `application.yml` 中数据库账号密码。
- [ ] （仅当前端独立运行时）安装 Node LTS，`npm install` 后再 `npm run dev`/`build`。
- [ ] `java -jar jsa.jar` 启动，浏览器访问 `http://localhost:8080` 验证。
