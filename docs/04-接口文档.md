# 接口文档（RESTful API）

> 项目：健身打卡系统（JSA）
> 基础路径：`/api`
> 数据格式：`application/json; charset=utf-8`
> 文档版本：v1.0
> 最后更新：2026-06-16

## 1. 通用约定

### 1.1 统一响应体

```json
{
  "code": 200,
  "message": "success",
  "data": { }
}
```

### 1.2 HTTP 状态码使用

| HTTP 状态码 | 使用场景 |
|-------------|----------|
| 200 OK | 请求成功（含查询、更新成功） |
| 201 Created | 资源创建成功（提交打卡） |
| 400 Bad Request | 参数错误、校验失败、业务非法操作 |
| 401 Unauthorized | 未登录 / token 无效 / 登录失败 |
| 403 Forbidden | 已登录但无权限（如普通用户访问管理接口） |
| 404 Not Found | 资源不存在 |
| 500 Internal Server Error | 服务端异常 |

### 1.3 业务错误码（data 中的 code）

| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 参数/业务错误 |
| 401 | 未登录或登录失败 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 1001 | 用户名或密码错误 |
| 1002 | 打卡记录不存在 |
| 1003 | 该记录非待审核状态，不可重复审核 |
| 500 | 服务器内部错误 |

### 1.4 认证

除登录接口外，所有接口需在请求头携带 token：

```
Authorization: Bearer <token>
```

token 由登录接口下发。需要管理员权限的接口额外要求 `role == ADMIN`，否则返回 403。

### 1.5 状态枚举

- 角色 `role`：`USER` | `ADMIN`
- 审核状态 `status`：`PENDING`（待审核）| `APPROVED`（通过）| `REJECTED`（驳回）

## 2. 接口一览

| 模块 | 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|------|
| 认证 | POST | `/api/auth/login` | 登录（用户/管理员通用） | 公开 |
| 运动项目 | GET | `/api/sports` | 获取运动项目列表 | 登录 |
| 打卡 | POST | `/api/checkins` | 提交打卡 | USER |
| 打卡 | GET | `/api/checkins/mine` | 查看本人打卡记录 | USER |
| 打卡 | GET | `/api/checkins` | 查看全部打卡记录 | ADMIN |
| 打卡 | PATCH | `/api/checkins/{id}/review` | 审核打卡（通过/驳回） | ADMIN |
| AI（预留） | POST | `/api/ai/advice` | 获取 AI 健身建议 | 详见 05 文档 |

## 3. 接口详情

### 3.1 登录

`POST /api/auth/login`

**请求体**

```json
{
  "username": "alice",
  "password": "123456"
}
```

**成功响应** `200`

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "b1f3c8e2-9a4d-4f2a-8e7c-1234567890ab",
    "user": {
      "id": 2,
      "username": "alice",
      "nickname": "爱丽丝",
      "role": "USER"
    }
  }
}
```

**失败响应** `401`（用户名或密码错误）

```json
{ "code": 1001, "message": "用户名或密码错误", "data": null }
```

> 前端根据返回的 `role` 决定跳转到用户中心还是管理后台。

---

### 3.2 获取运动项目列表

`GET /api/sports`

用于打卡页的项目下拉选择。

**成功响应** `200`

```json
{
  "code": 200,
  "message": "success",
  "data": [
    { "id": 1, "name": "跑步", "description": "有氧运动，建议每次 30 分钟以上" },
    { "id": 3, "name": "力量训练", "description": "哑铃、杠铃等器械训练" }
  ]
}
```

---

### 3.3 提交打卡

`POST /api/checkins`  权限：`USER`

**请求体**

```json
{
  "sportId": 1,
  "content": "晨跑 5 公里，状态不错"
}
```

> `userId`、`checkinTime`、`status` 均由服务端补充：用户从 token 解析，时间取服务端当前时间，状态默认 `PENDING`。前端不传。

**成功响应** `201`

```json
{
  "code": 200,
  "message": "打卡成功，等待审核",
  "data": {
    "id": 10,
    "sportId": 1,
    "sportName": "跑步",
    "content": "晨跑 5 公里，状态不错",
    "status": "PENDING",
    "checkinTime": "2026-06-16 08:30:00"
  }
}
```

**失败响应** `400`

```json
{ "code": 400, "message": "运动项目不存在", "data": null }
```

---

### 3.4 查看本人打卡记录

`GET /api/checkins/mine`  权限：`USER`

用户只能查看自己的记录，`userId` 从 token 解析。

**查询参数（可选）**

| 参数 | 类型 | 说明 |
|------|------|------|
| status | string | 按审核状态过滤：`PENDING`/`APPROVED`/`REJECTED` |

**成功响应** `200`

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 10,
      "sportName": "跑步",
      "content": "晨跑 5 公里，状态不错",
      "status": "PENDING",
      "checkinTime": "2026-06-16 08:30:00",
      "reviewTime": null
    }
  ]
}
```

---

### 3.5 查看全部打卡记录（管理员）

`GET /api/checkins`  权限：`ADMIN`

返回所有用户的打卡记录，包含用户名（联表查询）。

**查询参数（可选）**

| 参数 | 类型 | 说明 |
|------|------|------|
| status | string | 按状态过滤，常用于筛选 `PENDING` 待审核 |

**成功响应** `200`

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 9,
      "username": "bob",
      "nickname": "鲍勃",
      "sportName": "游泳",
      "content": "游了 1000 米",
      "status": "PENDING",
      "checkinTime": "2026-06-16 07:10:00",
      "reviewerNickname": null,
      "reviewTime": null
    }
  ]
}
```

---

### 3.6 审核打卡

`PATCH /api/checkins/{id}/review`  权限：`ADMIN`

对待审核记录执行通过或驳回。

**路径参数**：`id` — 打卡记录 id

**请求体**

```json
{
  "action": "APPROVE"
}
```

| 字段 | 取值 | 说明 |
|------|------|------|
| action | `APPROVE` | 审核通过，记录状态置为 `APPROVED` |
| action | `REJECT` | 审核驳回，记录状态置为 `REJECTED` |

> 服务端写入 `reviewerId`（当前管理员）与 `reviewTime`（当前时间）。

**成功响应** `200`

```json
{
  "code": 200,
  "message": "审核完成",
  "data": {
    "id": 9,
    "status": "APPROVED",
    "reviewTime": "2026-06-16 09:00:00"
  }
}
```

**失败响应**

- `404` 记录不存在：`{ "code": 1002, "message": "打卡记录不存在", "data": null }`
- `400` 非待审核状态：`{ "code": 1003, "message": "该记录已审核，不可重复操作", "data": null }`

## 4. RESTful 设计说明

| 设计点 | 说明 |
|--------|------|
| 资源命名 | 使用名词复数：`/checkins`、`/sports`。 |
| 方法语义 | 查询用 GET、创建用 POST、局部更新（审核）用 PATCH。 |
| 用户视角隔离 | `/checkins/mine` 表达"我的"子资源；`/checkins` 为管理员全量视角，靠权限区分。 |
| 审核为何用 PATCH | 审核是对记录 `status` 字段的局部状态变更，语义上是部分更新，故用 PATCH 而非 PUT。 |
| 无状态 | 身份信息由 token 携带，服务端不依赖会话页面状态。 |
